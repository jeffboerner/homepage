How to use: the Chrome extension should win over the native app, so to test the extension, quit the native TextExpander app first. The Chrome visual icon appearing on a field is the signal the extension is tracking it.

This mimics ServiceNow's Customer Work Form: top → workspace srcdoc iframe → record-form srcdoc iframe (both about:srcdoc, same-origin as parent). The nested iframes are built via JS to avoid srcdoc-escaping issues.

srcdoc iframe with src="about:srcdoc" — postMessage A/B

Same as a normal srcdoc frame, but the iframe also has src="about:srcdoc" set (some frameworks do this alongside srcdoc). That makes iframe.src non-empty and opaque, so the parent computes origin "null". Pre-fix prediction: old new URL("about:srcdoc") → origin "null" → relay dropped → no icon. Post-fix prediction: delivered via '*' → icon appears. Plain <input>, so this isolates the postMessage change from the contenteditable helper. This is the most likely real ServiceNow trigger.

Realistic combined case — opaque-src srcdoc + rich-text contenteditable

The closest thing to the real ServiceNow shape: an iframe with src="about:srcdoc" (opaque) whose field is a normal contenteditable. Click in and type ztext + your delimiter. Exercises both changes at once with a real caret: the postMessage relay must reach the opaque frame (icon), and expansion must land at the caret in rich text. Pre-fix: no icon (relay dropped). Post-fix: icon + expansion in place.

Nested srcdoc record form (2 levels deep)

Field A — normal rich text (control): caret focuses the contenteditable host, so document.activeElement is the host. Works before and after the fix.
Field B — inner-node focus (synthetic bug trigger): clicking focuses an inner <p tabindex="0">, so document.activeElement is the inner node, NOT the contenteditable host. This is the condition resolveEffectiveExpansionSurface exists for. Pre-fix: no TE icon (activeElement doesn't match the input selector). Post-fix: icon resolves to the host. (Synthetic — forces the code path; not a verified pixel match of ServiceNow's editor.)

about:blank iframe (filled via JS) — does the postMessage fix matter here?

Iframe created with src="about:blank" and populated by DOM injection (a common SPA editor pattern). Same-origin, so the content script IS injected — but the URL origin computes to "null". Pre-fix prediction: the parent drops the focusIn relay (origin "null" rejected) ⇒ no TE icon. Post-fix prediction: delivered via '*' ⇒ icon appears. This is the one case the postMessage change is predicted to help.
